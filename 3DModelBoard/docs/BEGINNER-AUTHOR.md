## 1. 일이 돌아가는 큰 그림

먼저, 이 게시판이 **어떻게 "글 쓰기"를 구현했는지** 를 가장 편한 비유로 정리합니다.

> **우체통 비유**
>
> - 운영자 계정 = 출판사
> - 운영자의 GitHub 리포 = 출판사 원본 원고철
> - 당신의 GitHub 계정 = 당신의 책상
> - 당신이 만든 **fork(포크)** = 당신 책상 위에 놓인 *출판사 원고철 사본*
> - **PR(풀 리퀘스트)** = 당신이 수정한 사본을 *"이렇게 바꿨습니다, 원본에도 반영해 주세요"* 라고 출판사에 보내는 우편
> - **PAT(토큰)** = 우편을 보낼 때 당신 이름으로 붙이는 *당신 전용 우표*

이 게시판은 다음 순서로 움직입니다.

1. 당신이 브라우저에서 모델 파일을 선택하고 **[🚀 PR로 공유하기]** 를 누릅니다.
2. 브라우저가 *당신의 PAT로 GitHub과 통신해서* 당신의 fork에 파일을 올립니다.
3. 그 fork에서 원본 리포로 "이 파일 넣어주세요"라는 PR이 자동으로 만들어집니다.
4. **운영자가 PR을 검토해서 승인(머지)** 하면, 그때 비로소 공개 게시판 목록에 당신 글이 뜹니다.

정리하자면, 당신의 글은 **"당신 브라우저 → 당신 GitHub fork → 운영자 리포"** 라는 세 단계를 거칩니다. 서버는 없습니다. 중간에 이 사이트의 자체 서버가 뭘 해주는 게 아니라, 전부 GitHub의 공식 서비스만 사용합니다.

> 💡 이게 왜 중요하냐: **"내가 누른 이 버튼이 뭘 하는지"** 를 알면 실수로 뭔가 잘못될까 걱정하지 않아도 됩니다. 당신 계정이 해낸 일은 `git log`에 다 남고, 운영자 허락이 없으면 원본에는 아무것도 반영되지 않습니다.

## 2. GitHub 계정 준비 (10분)

이미 GitHub 계정이 있다면 [3번으로 건너뛰세요](#3-이-게시판을-내-계정으로-복제하기-fork).

### 2-1. 계정 만들기

1. <https://github.com/signup> 열기.
2. 이메일 주소 입력. (이 이메일은 나중에 공개 설정이 있습니다 — 아래 *2-3* 참고)
3. 비밀번호 설정. *반드시* 다른 사이트에서 쓰지 않는 비밀번호로.
4. 사용자명(username) 설정. **이 이름이 공개됩니다.** 실명을 넣지 마세요. 취미용 닉네임 권장.
5. 퍼즐 인증 → 이메일로 온 코드 입력 → 완료.

### 2-2. 2단계 인증(2FA) 설정

**반드시 켜세요.** 2024년 이후 GitHub에서 필수화되었습니다.

- 오른쪽 위 프로필 아이콘 → **Settings** → **Password and authentication** → **Two-factor authentication**.
- 구글 Authenticator 앱이나 비슷한 TOTP 앱을 스마트폰에 설치.
- 화면의 QR코드를 앱으로 찍고, 앱에 뜬 6자리 숫자를 입력.
- *복구 코드(recovery codes)가 나오면 반드시 프린트하거나 메모해서 오프라인에 보관.* 폰이 고장나면 이 코드가 유일한 열쇠입니다.

### 2-3. 이메일 비공개 설정 (권장)

- Settings → **Emails** → **"Keep my email addresses private"** 체크.
- 이걸 켜두면 당신이 커밋한 기록에 진짜 이메일이 안 붙고 `123456+username@users.noreply.github.com` 같은 가짜 주소가 붙습니다.

> 🔒 이 게시판은 **공개 리포지토리** 기반입니다. 당신이 올린 닉네임·커밋 메타데이터는 *영구 공개*입니다. 실명·이메일·휴대폰 번호·주소가 어디에도 섞여 들어가지 않도록 주의.

## 3. 이 게시판을 내 계정으로 복제하기 (fork)

글을 쓰려면 **당신 계정에도 이 게시판 리포의 사본이 있어야** 합니다. 사본 = fork.

> 💡 **왜 사본이 필요한가요?** GitHub에서 다른 사람의 공개 리포에 직접 파일을 올릴 권한은 일반적으로 없습니다. 그래서 "내 사본에는 내가 마음대로 올릴 수 있다"는 구조를 쓰는 것입니다. 나중에 PR로 원본에 반영을 요청하게 됩니다.

### 3-1. fork 방법

1. 이 게시판의 원본 리포 페이지 열기. 주소는 운영자가 알려줍니다. (예: `https://github.com/원본계정/3DModelBoard`)
2. 오른쪽 위의 **[Fork]** 버튼 클릭.
3. "Owner"에 당신 계정이 이미 선택되어 있을 겁니다. 그대로 **[Create fork]**.
4. 몇 초 뒤 당신 계정의 새 리포(`https://github.com/당신계정/3DModelBoard`)로 자동 이동합니다. 끝.

### 3-2. 확인

이 두 가지가 보이면 OK:
- 주소가 `github.com/당신계정/3DModelBoard`
- 페이지 위쪽에 "forked from 원본계정/3DModelBoard" 라는 작은 문구

## 4. (운영자인 경우만) 내 계정에서 사이트 띄우기

당신이 **글을 올리는 쪽**이라면 이 단원은 건너뛰세요 — 운영자 URL을 그냥 사용하면 됩니다.

당신이 **이 게시판을 자기 이름으로 운영하고 싶다면**, fork한 내 리포가 GitHub Pages에서 서빙되게 해야 합니다.

### 4-1. GitHub Pages 켜기

1. 당신 리포(`https://github.com/당신계정/3DModelBoard`) → **Settings** 탭.
2. 왼쪽 사이드바 **Pages** 클릭.
3. Source: **Deploy from a branch** 선택.
4. Branch: **`main`** / Folder: **`/ (root)`** 선택 → **Save**.
5. 몇 분 기다리면 페이지 상단에 **Your site is live at `https://당신계정.github.io/3DModelBoard/`** 가 표시됩니다.

### 4-2. Branch protection (권장)

외부에서 들어온 PR을 확인하지 않고 자동으로 머지되지 않도록:
- Settings → **Branches** → **Add branch ruleset** 또는 **Add rule**.
- Branch name pattern: `main`.
- ✅ Require a pull request before merging
- ✅ Require approvals (1명 이상)
- ✅ Disallow force pushes.

## 5. 글을 올리려면 뭐가 필요한가 — PAT 이야기

당신의 브라우저가 GitHub에 "이 파일 올려주세요, 이 PR 만들어주세요"라고 말할 때, GitHub은 *"네가 정말 그 계정 주인이야?"* 를 확인하려 합니다. 이때 쓰는 열쇠가 **PAT (Personal Access Token, 개인 접근 토큰)** 입니다.

비밀번호 대신 쓰는 **"한정된 권한만 가진 일회용 긴 문자열"** 이라고 보면 됩니다. 유출되어도 피해 범위가 작고, 언제든 즉시 폐기할 수 있습니다.

### 5-1. 이 게시판에서 PAT를 다루는 방식

이 게시판 코드는 당신의 PAT를

- **어디에도 저장하지 않습니다.** 서버가 없으니 서버에 보낼 일도 없고, 쿠키/로컬스토리지에도 기록하지 않습니다.
- **당신 브라우저의 임시 메모리** 에만 잠시 뒀다가, 탭을 닫거나 새로고침하면 즉시 지워집니다.

결과적으로 글을 쓸 때마다 **PAT를 다시 붙여넣어야 합니다.** 불편해 보이지만, 이게 일부러 설계된 안전 장치입니다.

### 5-2. 어떤 권한을 줘야 하나

이 게시판이 당신의 PAT로 수행하는 작업은 딱 두 가지뿐입니다.
- 당신 fork에 파일을 커밋하기 → **Contents: Read and write**
- 당신 fork에서 원본으로 PR 만들기 → **Pull requests: Read and write**

*더 많은 권한을 주지 마세요.* 특히 "모든 리포지토리에 접근" 같은 건 절대 금지.

## 6. PAT 발급받기 (한 단계씩)

화면 경로가 시간이 지나며 살짝 바뀔 수 있어서, 단계 이름을 정확히 따라가세요.

### 6-1. 토큰 생성 화면으로

1. GitHub 오른쪽 위 프로필 아이콘 → **Settings**.
2. 왼쪽 사이드바 맨 아래 **Developer settings** 클릭.
3. **Personal access tokens → Fine-grained tokens** 클릭. (*"Tokens (classic)" 이 아니라 "Fine-grained" 를 꼭 고르세요. 권한 제어가 세밀하고 안전합니다.*)
4. **[Generate new token]** 버튼 클릭.

### 6-2. 토큰 설정

화면 항목을 이렇게 채웁니다.

| 항목 | 무엇을 넣나 |
| --- | --- |
| **Token name** | `3dmodelboard-post` 같이 알아볼 수 있는 이름 |
| **Expiration** | 7일 ~ 90일 권장. 짧게 잡을수록 안전. 기간이 지나면 새로 만듭니다 |
| **Description** | (선택) `3DModelBoard 게시용` 등 |
| **Repository access** | **Only select repositories** 선택 → 당신의 **`3DModelBoard` fork 딱 한 개만** 골라 추가 |
| **Repository permissions** | 아래 표대로 |

| 권한 항목 | 설정값 | 이유 |
| --- | --- | --- |
| **Contents** | **Read and write** | 게시물 파일 커밋 |
| **Pull requests** | **Read and write** | 원본에 PR 만들기 |
| **Metadata** | Read-only (자동 포함) | 어쩔 수 없는 기본값 |
| 그 외 모든 항목 | 건드리지 말기 (기본값 No access) | 권한은 최소로 |

화면 아래 **[Generate token]** 클릭.

### 6-3. 토큰 복사 — *딱 한 번만 보여줍니다*

`github_pat_...` 으로 시작하는 긴 문자열이 나옵니다. **이 화면을 떠나면 다시는 볼 수 없습니다.** 안전한 곳에 잠깐 복사해 두세요(파일로 저장하지 말고 클립보드에 두는 정도).

> 🔒 절대 하지 말 것:
> - 이 토큰을 이메일·카톡·슬랙에 붙여넣기
> - 스크린샷을 남겨 공유 폴더에 넣기
> - GitHub 이슈 댓글이나 PR 본문에 섞기
>
> *잘못 유출되었다 싶으면* 즉시 토큰 목록으로 돌아와 **[Revoke]** 버튼을 누르세요. 기존 토큰이 즉시 무효화되고, 새 토큰을 다시 발급하면 됩니다.

## 7. 설정 페이지 입력

이 게시판의 브라우저 UI로 돌아와서 상단 메뉴 **설정** 또는 `settings.html` 로 갑니다.

### 7-1. 리포지토리 카드

| 필드 | 넣을 값 |
| --- | --- |
| **Upstream Owner** | 당신이 PR을 보낼 대상, 즉 **운영자의 GitHub 계정명**. 예: `someuser` |
| **Repository** | `3DModelBoard` (원본 리포 이름. 별도 포크명이 아니라 *원본* 이름) |
| **Base branch** | `main` (특별한 경우가 아니면 바꾸지 마세요) |

**[저장]** 버튼 클릭.

### 7-2. 작성자 카드

- **작성자 이름**: 닉네임. **실명 금지** — 공개됩니다.

### 7-3. PAT 카드

- 방금 복사한 `github_pat_...` 을 붙여넣고 **[연결 테스트]** 클릭.
- 토큰이 유효하면 **"인증됨: 당신계정"** 이라고 녹색 메시지가 뜹니다.
- 실패하면 → 토큰 오타 가능성, 기간 만료, 권한 부족 중 하나. [12-문제해결](#12-실수문제가-생겼을-때) 참고.

> 💡 새 탭을 열거나 새로고침하면 PAT가 사라집니다. 그때마다 여기로 돌아와 다시 붙여넣으세요.

## 8. 모델 올리기 — 글쓰기 페이지

상단 **[+ 새 모델 올리기]** 또는 `submit.html`.

### 8-1. 모델 파일 (`.vrm` / `.pmx` / `.pmd`)

- 파일을 박스 위에 끌어다 놓거나 박스를 클릭해 선택.
- 크기 제한:
  - **PR로 공유**: 25MB 이하만 가능
  - **로컬 저장**: 100MB 까지
- 업로드하자마자 오른쪽 박스에 *미리보기 3D 렌더링* 이 뜹니다.
- 크기가 25MB 를 넘으면 화면에 경고가 뜨고 **[PR로 공유하기]** 버튼이 비활성화됩니다(로컬 저장은 여전히 가능).

> 🚫 **저작권 유의** — 다른 사람이 만든 모델을 허락 없이 공개 리포에 올리지 마세요. MMD·VRM 모델들은 배포 조건(재배포 허가 여부, 상업 이용 금지 등)이 제각각입니다. 재배포 가능한 모델만 올리세요.

### 8-2. 제목 · 작성자 · 태그 · 설명

- **제목**: 필수. 120자까지.
- **작성자**: 비워두면 `anonymous`.
- **태그**: 쉼표로 구분(`vrm, anime, cute`), 최대 8개.
- **설명**: 2,000자까지.

모든 값이 **영구 공개됩니다.** 개인정보는 빼세요.

### 8-3. 썸네일

- 기본은 **자동 스냅샷** — 미리보기 첫 프레임이 PNG로 잡힙니다.
- **📷 다시 찍기** — 미리보기에서 모델을 원하는 각도로 돌린 뒤 눌러 재캡처.
- **🖼 이미지 업로드** — 직접 PNG/JPG/WebP 고르기. 자동으로 정사각형으로 잘립니다.

| --- | --- |
| **없음** (기본) | 뷰어 배경은 기본 어두운 패널 색(#1a1a2e) |
| **이미지 업로드** | 당신이 고른 PNG/JPG/WebP 이미지가 뷰어 배경으로 깔림. PR 시 **리포에 업로드**됨 |
| **외부 URL** | 뷰어가 해당 HTTPS 주소에서 이미지를 *실시간 로드* 해서 배경으로 깖 |

### 9-1. 이미지 업로드 모드

- **[🏞 배경 이미지 선택]** 버튼 → 로컬 PC의 이미지 파일 선택.
- 제한: 최대 **5MB**, 확장자 `.png` / `.jpg` / `.jpeg` / `.webp`.
- 선택하면 즉시 미리보기 뷰어와 하단 배경 미리보기 박스에 반영됩니다.
- PR로 공유 시 `posts/<당신-게시물-id>/background.확장자` 로 리포에 올라갑니다. 로컬 저장 시엔 IndexedDB에 Blob으로 보관.

### 9-2. 외부 URL 모드

- URL 입력 칸에 `https://` 로 시작하는 이미지 주소를 입력.
- 허용되는 것:
  - `https://` 프로토콜만 (**http:// 는 거부**)
  - 공개 호스트만 (**localhost, 127.0.0.1, 사설망 10.x/192.168.x/172.16-31.x 는 거부**)
- 주소 형식이 맞지 않으면 미리보기에 반영되지 않습니다.

> ⚠ **프라이버시 경고**: 외부 URL을 쓰면 *게시물을 여는 모든 방문자의 브라우저가 해당 외부 서버로 이미지 요청을 한 번 보냅니다.* 이건 IP 주소·User-Agent 정도가 그 서버 로그에 남을 수 있다는 뜻입니다. Referrer는 보내지 않도록 사이트가 막고 있으니 어느 사이트에서 왔는지는 안 알려지지만, 접속 자체는 기록될 수 있습니다. 신뢰할 수 있는 CDN이나 자기 서버만 쓰세요.

> 🧭 **선택 가이드**:
> - 운영자 리포의 저장 용량을 아끼고 싶다 → 외부 URL
> - 방문자 프라이버시를 더 중시한다 → 이미지 업로드
> - 잘 모르겠다 → **이미지 업로드** 를 권장

### 9-3. 배경 없애기

- 라디오 버튼을 **없음** 으로 되돌리면 배경이 사라지고 기본 어두운 패널로 보입니다.

## 10. "PR로 공유하기"를 누르면 무슨 일이 일어나는가

**[🚀 PR로 공유하기]** 버튼을 누르면 브라우저가 밑에서 이런 일들을 순서대로 합니다. 각 단계는 로그 창에 실시간 표시됩니다.

```
1. 토큰 확인           → PAT가 유효하고, 당신 이름이 뭔지 확인
2. fork 보장           → 당신 계정에 fork가 없으면 방금 만든다
3. 기준 커밋 가져오기  → main 브랜치의 최신 SHA(해시)
4. 새 브랜치 만들기    → post/<글-아이디>-<시각>
5. 모델 파일 업로드    → posts/<id>/model.vrm (등)
6. 썸네일 업로드       → posts/<id>/thumbnail.png
7. (배경이 있으면)     → posts/<id>/background.<ext>
8. meta.json 업로드    → 제목·작성자·태그·포맷·배경 정보
9. posts.json 갱신     → 전체 글 목록에 당신 글 추가
10. PR 생성           → "이 브랜치 병합해 주세요" 요청을 원본 리포로 보냄
```

각 단계는 GitHub API에 대한 HTTP 요청 한 번입니다. 이상이 없으면 10~30초 안에 끝나고, 로그창 마지막에 **PR 주소 링크** 가 뜹니다. 링크를 클릭하면 실제 GitHub PR 페이지로 갑니다.

> 💡 이 전 과정이 **당신 브라우저**에서만 실행됩니다. 이 게시판에 따로 서버가 돌아가는 게 아니라 GitHub의 공식 API가 전부 일을 합니다.

## 11. PR 이후 — 운영자 승인 기다리기

PR은 **"운영자가 검토 후 머지할 때까지 아무 일도 일어나지 않은 상태"** 입니다.

### 11-1. 공개 순서

1. 운영자가 GitHub에서 당신의 PR을 열어 봅니다.
2. 모델이 저작권에 문제가 없고, 개인정보가 안 섞였고, 크기/형식이 맞는지 확인.
3. 괜찮으면 **Merge** 버튼. 이 순간부터 `main` 브랜치에 당신 파일이 들어갑니다.
4. GitHub Pages가 자동으로 재배포(보통 1~3분). 완료 후에는 공개 주소의 목록 페이지에 당신 글이 뜹니다.
5. *반대로 운영자가 거절*하거나 수정 요청하면 그 내용이 PR 코멘트로 달립니다.

### 11-2. 당신이 할 수 있는 일

- PR 주소를 북마크해두면 운영자 반응을 실시간으로 볼 수 있습니다.
- 수정 요청이 오면 → 이 게시판에서 **다시 올리지 마세요**. 중복 PR이 생깁니다. 대신 GitHub 웹에서 해당 파일을 수정하거나, 간단하면 기존 PR을 닫고 새로 올리세요.
- 당신이 마음이 바뀌어 글을 취소하고 싶으면 → PR 페이지 아래 **Close pull request** 버튼.

## 12. 실수/문제가 생겼을 때

### 12-1. 로그에 뜨는 에러 코드별 대응

| 로그에 나오는 메시지 | 원인 | 해결 |
| --- | --- | --- |
| `GitHub API 오류 (401): Bad credentials` | PAT 오타/만료 | 설정 페이지에서 PAT 재입력. 만료 시 6번으로 돌아가 새로 발급 |
| `GitHub API 오류 (403): rate limit` | 1시간당 요청 한도 초과 | 최대 1시간 대기 후 재시도 |
| `GitHub API 오류 (404)` (fork 단계) | Upstream Owner/Repository 이름 오타 | 설정 페이지에서 철자 다시 확인 |
| `GitHub API 오류 (422)` (PR 생성) | 같은 브랜치로 이미 PR이 열려 있음 | GitHub에서 기존 PR 먼저 닫거나 머지 |
| `GitHub API 오류 (403): Resource not accessible` | 토큰 권한 부족 | 6-2로 돌아가 권한 재확인(Contents·Pull requests 둘 다 write) |
| `파일 헤더 검증 실패` | 확장자만 바꾼 가짜 파일 | 정상적인 `.vrm`/`.pmx`/`.pmd` 파일만 사용 |
| `25MB 초과 → PR 불가` | 모델이 너무 큼 | 로컬 저장으로 전환하거나 모델을 축소/최적화 |

### 12-2. 중간에 실패하면 내 fork 브랜치는 어떻게 되나

업로드가 절반만 되고 PR 생성에서 실패하는 경우, **당신 fork에는 이미 일부 파일이 커밋된 브랜치가 남아 있습니다.** 그대로 두면 다음 PR에 섞이거나 혼란을 일으킵니다.

정리 방법:
1. GitHub 웹에서 `https://github.com/당신계정/3DModelBoard` 열기.
2. 브랜치 드롭다운에서 `post/...` 로 시작하는 브랜치 선택.
3. 오른쪽 **Delete branch** 또는 브랜치 페이지에서 휴지통 아이콘 클릭.
4. 이 게시판에서 다시 **[PR로 공유하기]** 누르면 새 브랜치로 재시도됩니다.

### 12-3. "내 PR에 이상한 게 들어 있어요"

PR 페이지의 **Files changed** 탭에서 추가된 파일 전부를 눈으로 확인하세요. 만약 개인정보가 섞여 있거나 의도하지 않은 파일이 보이면:
- PR 페이지 하단 **Close pull request** 로 닫기.
- 원인 된 파일을 당신 fork에서 지우고 새로 시도.

### 12-4. PAT를 잃어버렸어요

- 토큰 자체는 한 번만 보여주므로 *잃어버린 건 복구 불가능*.
- 걱정하지 마세요. 그냥 GitHub Settings → Developer settings → Personal access tokens → 문제의 토큰 **Revoke** 한 다음, 새 토큰을 발급받으면 됩니다.

### 12-5. 누군가 내 PAT를 훔쳐봤을지도

1. 즉시 해당 토큰 Revoke.
2. 그 토큰으로 생성된 PR / 브랜치 / 커밋 전부 확인. 모르는 게 있으면 닫기·되돌리기.
3. (이 게시판에선 해당 없지만 혹시) 해당 토큰이 다른 리포에도 권한을 가졌다면 — 토큰 생성 시 "Only select repositories" 를 골랐으면 *이 게시판 fork 하나만* 영향이니 괜찮습니다.

## 13. 안전하게 쓰기 위한 7가지 규칙

1. **실명·이메일·전화번호·주소를 절대 입력하지 마세요.** 공개 리포는 영구 기록입니다.
2. **PAT는 Fine-grained 로만, 이 fork 하나에만 권한.** 수명은 짧게.
3. **PAT를 이메일·메신저·스크린샷으로 공유하지 마세요.** 의심되면 즉시 Revoke.
4. **저작권이 확인되지 않은 모델을 올리지 마세요.** 상업·2차 배포 금지 모델이 많습니다.
5. **업로드 전에 미리보기에서 한 번 확인.** 엉뚱한 파일을 고른 경우 바로 알 수 있습니다.
6. **"내 브라우저에 저장"과 "PR로 공유하기"를 헷갈리지 마세요.** 전자는 비공개, 후자는 전세계 공개.
7. **외부 URL 배경은 신뢰할 수 있는 CDN만.** 추적용 도트 픽셀이 숨어있을 수 있는 사설 URL은 피하세요.

## 14. 용어 사전

| 용어 | 풀이 |
| --- | --- |
| **GitHub** | 프로그래머들이 파일을 공유·버전 관리하는 가장 큰 온라인 서비스. 무료 계정으로도 공개 리포를 만들 수 있음. |
| **리포(지토리) / repo** | GitHub에서 프로젝트가 담기는 폴더 같은 단위. |
| **fork(포크)** | 다른 사람의 리포를 *내 계정으로 복제*하는 것. |
| **branch(브랜치)** | 리포 안의 "작업 분기선". 이 게시판은 글마다 `post/...` 이름의 브랜치를 만든다. |
| **commit(커밋)** | 파일 변경을 영구 기록하는 단위. 스냅샷 한 컷. |
| **PR (Pull Request)** | "내 fork의 이 브랜치를 원본 리포에 합쳐 주세요"라는 요청. 코드 리뷰 수단. |
| **merge(머지)** | 운영자가 PR을 승인해 원본에 반영하는 행위. |
| **PAT (Personal Access Token)** | 비밀번호 대신 쓰는 API용 열쇠. 권한·유효기간을 제한해 발급. |
| **Fine-grained PAT** | "이 리포에만, 이 권한만" 이라고 세밀하게 잘라 발급하는 새 방식의 PAT. Classic 보다 안전. |
| **main 브랜치** | 이 리포의 기본 브랜치. 여기 머지되면 GitHub Pages에서 공개됨. |
| **GitHub Pages** | GitHub이 무료로 제공하는 정적 웹사이트 호스팅. `당신계정.github.io/리포이름/` 형식. |
| **CSP** | Content Security Policy — 브라우저가 어떤 출처의 자원(스크립트·이미지)만 허용할지 정한 규칙. 이 사이트는 엄격하게 설정되어 있음. |
| **CDN** | Content Delivery Network — 이미지·스크립트를 빠르게 서빙하는 글로벌 서버망. 외부 URL 배경으로 쓰기 좋은 대상. |
| **Blob** | 브라우저가 파일을 메모리에 담고 있을 때의 형태. 이 사이트가 로컬 저장 시 배경 이미지를 Blob으로 보관. |
| **IndexedDB** | 브라우저 내부의 큰 데이터 저장소. 로컬 게시물이 여기에 들어감. |
| **Rate limit** | GitHub API가 1시간 단위로 계정당 요청 횟수를 제한하는 것. |
| **Revoke** | 이미 발급된 토큰·권한을 즉시 무효화하는 것. |
